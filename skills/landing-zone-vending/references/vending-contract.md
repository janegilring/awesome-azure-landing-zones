# Vending contract — input schema and trigger

A concrete, reusable specification for the **vending contract** referenced in
[Scenario 1](../../../docs/scenarios.md#scenario-1--self-service-subscription-vending-with-a-request-intake).
The contract is the boundary between the **vending backend** (this skill / the `landing-zone-vending`
agent) and the **self-service intake** (a web portal or an AI agent). It has two parts:

1. **Input schema** — the fields a requester must provide.
2. **Trigger mechanism** — how a validated request starts the deployment.

A frontend implements this contract: web form fields or an AI agent's tool parameters map 1:1 to the
input schema, and "submit" performs the trigger. Treat this file as a template — adjust field names,
allowed values, and validation to your platform's naming and policy.

## Part 1 — input schema

| Field | Type | Required | Validation / allowed values | Maps to (lz-vending) |
| :--- | :--- | :--- | :--- | :--- |
| `workloadName` | string | yes | `^[a-z][a-z0-9-]{2,23}$` (kebab-case, 3–24 chars) | subscription alias + naming |
| `environment` | enum | yes | `dev` \| `test` \| `prod` | tag + management-group placement |
| `owner` | string (UPN) | yes | valid Entra user/group; resolvable | `subscriptionOwnerId` / RBAC |
| `costCenter` | string | yes | matches finance code pattern, e.g. `^CC[0-9]{4,6}$` | tag `CostCenter` + budget |
| `businessUnit` | string | yes | from an approved list | tag `BusinessUnit` |
| `dataClassification` | enum | yes | `public` \| `internal` \| `confidential` \| `restricted` | policy scope / mg placement |
| `connectivityModel` | enum | yes | `none` \| `hub-spoke` \| `vwan` | VNet peering / vWAN connection |
| `region` | enum | yes | from approved regions, e.g. `westeurope`, `norwayeast` | resource region |
| `vNetAddressSpace` | string (CIDR) | conditional | required if `connectivityModel != none`; valid, non-overlapping (IPAM-checked) | spoke VNet address space |
| `budgetAmount` | number | yes | `> 0`; currency from platform default | preliminary `budget` |
| `managementGroupId` | string | derived | derived from `environment` + `dataClassification`; not user-supplied | target management group |
| `additionalTags` | map<string,string> | optional | keys/values within policy-allowed tag rules | resource tags |
| `requestId` | string (UUID) | system | generated by the intake; trackable | correlation / reporting |
| `requestedBy` | string (UPN) | system | the authenticated requester (not free-text) | audit |

Notes:

- **Derived, not requested.** `managementGroupId` is computed by the backend from `environment` and
  `dataClassification` so requesters cannot place a subscription in the wrong scope. The intake must
  **not** expose it as an editable field.
- **System-set, not requested.** `requestId` and `requestedBy` come from the intake/identity layer,
  never free-text input — this is an integrity control.
- **Validate early.** Run schema, allowed-value, and **IP-range availability** (IPAM) checks at
  intake time; reject before a parameter file is ever created.

### JSON Schema (intake validation)

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Landing zone vending request",
  "type": "object",
  "additionalProperties": false,
  "required": [
    "workloadName", "environment", "owner", "costCenter", "businessUnit",
    "dataClassification", "connectivityModel", "region", "budgetAmount"
  ],
  "properties": {
    "workloadName":      { "type": "string", "pattern": "^[a-z][a-z0-9-]{2,23}$" },
    "environment":       { "enum": ["dev", "test", "prod"] },
    "owner":             { "type": "string", "format": "email" },
    "costCenter":        { "type": "string", "pattern": "^CC[0-9]{4,6}$" },
    "businessUnit":      { "type": "string" },
    "dataClassification":{ "enum": ["public", "internal", "confidential", "restricted"] },
    "connectivityModel": { "enum": ["none", "hub-spoke", "vwan"] },
    "region":            { "enum": ["westeurope", "norwayeast"] },
    "vNetAddressSpace":  { "type": "string", "pattern": "^([0-9]{1,3}\\.){3}[0-9]{1,3}/[0-9]{1,2}$" },
    "budgetAmount":      { "type": "number", "exclusiveMinimum": 0 },
    "additionalTags":    { "type": "object", "additionalProperties": { "type": "string" } }
  },
  "allOf": [
    {
      "if":   { "properties": { "connectivityModel": { "not": { "const": "none" } } } },
      "then": { "required": ["vNetAddressSpace"] }
    }
  ]
}
```

## Part 2 — trigger mechanism

The validated request becomes **one subscription parameter file** and starts deployment. Pick one of
two patterns; the intake performs the corresponding action.

### Pattern A — Gitflow (parameter file + pull request) — recommended

1. Intake renders the request into a parameter file at a deterministic path, e.g.
   `subscriptions/<environment>/<workloadName>.bicepparam` (or `.tfvars` / `.json`).
2. Intake commits it to a **new branch** and opens a **pull request** into `main`.
3. PR validation runs (lint, schema, IP-range availability, optional manual gate).
4. **Merge to `main` triggers the deployment pipeline** that runs the sub-vending module.

This keeps every landing zone request auditable and reviewable in Git, and keeps the IaC
platform-owned. The intake needs permission only to open a PR — not to deploy.

### Pattern B — pipeline dispatch (API)

1. Intake calls the deployment pipeline directly via a dispatch API:
   - GitHub Actions `workflow_dispatch` (REST: `POST /repos/{owner}/{repo}/actions/workflows/{id}/dispatches`), or
   - Azure DevOps pipeline run REST API.
2. The pipeline materializes the parameter file and runs the module.

Use when a PR-per-request is too heavy, but you lose the Git review trail unless the pipeline still
commits the rendered parameter file for audit.

> **Identity:** the intake authenticates to Git/pipeline with a **least-privilege, secretless**
> identity (OIDC / workload identity federation). The deployment pipeline itself uses a separate
> managed identity scoped to create subscriptions and configure them — never embedded secrets.

### Example rendered parameter file (Bicep)

```bicep
using 'br/public:avm/ptn/lz/sub-vending:<version>'

param subscriptionAliasName    = 'contoso-payments-prod'
param subscriptionDisplayName  = 'Contoso Payments (prod)'
param subscriptionBillingScope = '/providers/Microsoft.Billing/billingAccounts/.../billingProfiles/.../invoiceSections/...'
param subscriptionManagementGroupId = 'mg-corp-confidential' // derived from environment + dataClassification
param virtualNetworkEnabled    = true
param virtualNetworkAddressSpace = ['10.64.12.0/24']
param virtualNetworkResourceGroupName = 'rg-contoso-payments-network'
param virtualNetworkLocation   = 'norwayeast'
param hubNetworkResourceId     = '/subscriptions/.../virtualNetworks/vnet-hub-norwayeast'
param roleAssignments          = [
  {
    principalId: '<owner-group-object-id>'
    roleDefinitionIdOrName: 'Owner'
    relativeScope: ''
  }
]
param tags = {
  Environment: 'prod'
  CostCenter: 'CC10472'
  BusinessUnit: 'Retail Banking'
  DataClassification: 'confidential'
  Application: 'Payments API'
  Tier: '1'
}
```

> The exact module parameters depend on the `bicep-lz-vending` / `terraform-lz-vending` version.
> Confirm against the current module schema before use (Bicep: `bicep` MCP `list_avm_metadata`;
> Terraform: the Registry tools).

## How a frontend consumes this contract

- **Web portal (Container Apps / App Service):** the JSON Schema drives form fields and client/server
  validation; "submit" performs Pattern A (open PR) or Pattern B (dispatch).
- **AI intake (Foundry agent):** the schema becomes the agent's tool/function parameter definition;
  the agent gathers values conversationally, validates against the schema, then calls a tool that
  performs the trigger.
